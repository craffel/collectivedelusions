\section{Experimental Details and Hyperparameters}
\label{sec:appendix_details}

In this appendix, we provide the complete architectural and optimization details for our task-specific experts, as well as the calibration and evaluation configurations used across all evaluated model merging and routing methods. This is provided to ensure full reproducibility of our empirical deconstruction.

\subsection{Routing Calibration Protocol}
For all dynamic weight-routing methods requiring calibration (OFS-Tune, Linear Router, QWS-Merge, BL-Router, and GLS-Router), we optimize their trainable routing parameters on a tiny, balanced calibration split of $64$ total samples ($16$ random samples per task). Crucially, to isolate the routing architecture as the single active variable and eliminate any variance from calibration set sampling, all data extraction and routing head initializations are conducted under a strictly controlled, fixed-seed environment (random seed 42). This guarantees a perfectly identical calibration partition and optimization trajectory across all benchmarked methods, ensuring full reproducibility.
The routing heads and layer-wise amplitudes are updated for exactly $100$ optimization steps using the Adam optimizer with a learning rate of $1\times 10^{-2}$. During each step, the loss is the joint cross-entropy loss across all four tasks evaluated on their respective expert heads.

\subsection{Evaluation Protocols}
\textbf{Homogeneous Evaluation:} We construct a balanced test suite consisting of $1000$ independent, randomly sampled images from each task's test partition. The joint mean accuracy is evaluated by passing these task images in homogeneous batches and measuring top-1 accuracy.

\textbf{Heterogeneous Evaluation:} To measure dynamic task adaptation, we shuffle all $2000$ test images ($500$ per task) into a single interleaved stream. This stream is processed in batch sizes of $B \in \{1, 16, 256\}$. At each step, the dynamic routing coefficients are computed dynamically for the incoming batch using only the current batch's pooled representations.

\subsection{AdaMerging Baseline Implementation \& Optimization Specifications}
We implement AdaMerging in its standard unsupervised Test-Time Adaptation (TTA) configuration to serve as a comparison baseline. Unlike offline-calibrated dynamic routing methods, AdaMerging optimizes its merging coefficients dynamically at test-time on the incoming test stream.
\begin{itemize}
    \item \textbf{Formulation:} Given learned raw log-coefficients $\tilde{C} \in \mathbb{R}^K$, task merging coefficients are computed using a Softmax activation: $C_k = \text{Softmax}(\tilde{C})_k$. The dynamically merged weight at layer $l$ is assembled as: $W_{merged}^{(l)} = W_{base}^{(l)} + \sum_{k=1}^K C_k V_k^{(l)}$.
    \item \textbf{Entropy Minimization Objective:} Because labels are unavailable at test-time, AdaMerging is optimized in an unsupervised manner by minimizing the Shannon entropy of the model's prediction probabilities on the incoming test batch:
    \begin{equation}
        \mathcal{L}_{entropy}(x) = - \frac{1}{|x|} \sum_{i=1}^{|x|} \sum_{c=1}^{C} p(c | x_i) \log p(c | x_i)
    \end{equation}
    where $p(c | x_i)$ is the prediction probability for class $c$ on sample $x_i$, computed using the dynamically merged model.
    \item \textbf{Optimization Hyperparameters:}
    \begin{itemize}
        \item \textbf{Optimizer:} Adam with a learning rate of $\eta = 1\times 10^{-2}$ and default momentum parameters ($\beta_1 = 0.9, \beta_2 = 0.999$).
        \item \textbf{Test-Time Optimization Steps:} Exactly $1$ gradient update step is performed on each incoming test batch before making predictions, adhering to standard single-pass test-time adaptation settings to prevent excessive latency.
        \item \textbf{Coefficient Constraints:} To prevent task coefficients from expanding into destructive parameter regions, the Task Arithmetic coefficients are bounded to a maximum scale of $0.3$ during optimization.
    \end{itemize}
\end{itemize}

\subsection{Visualizing Heterogeneity Collapse via Batch-Averaged Routing Coefficients}
\label{sec:appendix_heterogeneity_collapse}

In Section 4.4, we discuss the mathematical batch-averaging bottleneck that occurs when dynamic weight-routing models process heterogeneous streams under varying batch sizes $B$. To make this phenomenon concrete and provide a rigorous visualization, we report the step-by-step routing coefficients computed across a segment of the shuffled heterogeneous stream for $B=1$ vs.\ $B=256$.

When the stream is processed with a batch size of $B=1$, the input representation $z(x) \in \mathbb{R}^{1 \times D}$ consists of a single isolated task sample (e.g., an SVHN digit). The routing head computes highly specialized, peaky coefficients that direct the merged parameters towards that specific expert. For example:
\begin{itemize}
    \item \textbf{Step $t$ (SVHN input):} $\alpha = [0.01, 0.01, 0.02, 0.26]$ (routing $87\%$ of the weight-routing budget to the SVHN task vector).
    \item \textbf{Step $t+1$ (MNIST input):} $\alpha = [0.27, 0.01, 0.01, 0.01]$ (routing $90\%$ of the budget to the MNIST task vector).
\end{itemize}
This high-amplitude dynamic switching allows the model to process each sample at its specialized peak accuracy (e.g., $96.00\%$ SVHN, $97.10\%$ MNIST).

When the batch size scales to $B=256$, the input batch contains an evenly distributed, heterogeneous mixture of approximately $64$ samples from each of the $4$ tasks. Since the dynamic merging coefficients must be collapsed across the batch to assemble a single set of shared weights for the forward pass, the sample-level probabilities are averaged:
\begin{equation}
    \bar{\alpha}_k = \frac{1}{B} \sum_{b=1}^{256} \alpha_{k, b} \approx \frac{1}{4} \sum_{j=1}^4 \alpha_{k, \text{task}_j}
\end{equation}
Because the batch is highly representative of the global stream distribution, the batch mean collapses statically to a flat, uniform distribution:
\begin{equation}
    \bar{\alpha} \approx [0.075, 0.075, 0.075, 0.075] \quad (\text{summing to the budget cap of } 0.30)
\end{equation}
Across successive steps, as the stream is sliced into independent batches of $B=256$, the Central Limit Theorem guarantees that the task proportions remain highly stable. Consequently, the routing head ceases to exhibit dynamic input-dependent variation, producing nearly identical, flat coefficients at every single step. In this regime, the dynamic routing head collapses entirely to behave as a static, non-adaptive Uniform Merge (Task Arithmetic), losing its local domain-specialization benefits. This mathematical bottleneck highlights a fundamental physical limitation of weight-space routing on heterogeneous streams: dynamic routing is highly effective under small, sequential batch regimes, but degenerates to static compromise configurations as the inference batch size scales up.

\textbf{Analysis of Calibration Complexity:} 
As shown in Table~\ref{tab:complexity_comparison}, both QWS-Merge and our proposed \textbf{BSigmoid-Router} require extremely minimal offline calibration times (1.4 seconds and 1.1 seconds, respectively). This is because the calibration set consists of only 64 samples and is optimized for just 100 steps. In contrast, while AdaMerging requires zero offline calibration, it shifts the entire optimization burden to test-time.

\textbf{Analysis of Inference Latency \& the TTA Bottleneck:}
The inference latency results expose a massive, critical bottleneck for online Test-Time Adaptation. At a batch size of $B=1$, AdaMerging requires \textbf{482.0 ms} to process a single batch. This represents an astronomical \textbf{26$\times$ latency overhead} compared to our BSigmoid-Router (18.5 ms). 
This severe delay is caused by AdaMerging's online optimization design: for each incoming test batch, it must perform 20 optimization steps. In each step, it clones the base model parameters, combines them with task vectors in weight space, runs a full forward pass through the backbone, computes prediction entropy, performs backpropagation to obtain gradients with respect to the task coefficients, and executes an Adam optimizer step. This amounts to running 21 parameter aggregations, 21 full forward passes, and 20 backward passes per batch. 

In stark contrast, our \textbf{BSigmoid-Router} runs as a pure forward pass with zero active optimization, zero backward passes, and zero latency overhead from test-time gradient descent. Its routing head is a highly compact, parameter-efficient single linear projection layer (772 parameters) followed by independent Sigmoid activations, executing the dynamic routing policy in only a few microseconds. 

Detailed hardware profiling reveals that while a standard static forward pass (Uniform Merge) on pre-merged parameters takes only 3.5 ms, our dynamic routers require 18.5 ms. Under deep profiling of the dynamic routing forward loop, we find that the routing head's linear projection and activation take less than 0.05 ms. The remaining 14.9 ms of overhead is entirely consumed by PyTorch memory management—specifically, copying and cloning the 5.7M parameters of the base model and performing element-wise floating-point additions to superimpose the active task vectors across 12 layers. This indicates that approximately 80\% of our dynamic router's inference time in this PyTorch prototype is spent on tensor buffer duplication and memory transfer rather than neural computation.

In production deployments, this memory overhead can be almost entirely bypassed. Techniques such as compile-time model graph optimization (e.g., \texttt{torch.compile(mode="max-autotune")} or TensorRT), kernel fusion (which fuses parameter cloning, scaling, and addition into a single, highly parallelized CUDA kernel), or hardware-accelerated parameter-space Mixture-of-Experts (MoE) block gating layouts can eliminate physical parameter duplication. This profiling confirms that the BSigmoid-Router is not bottlenecked by mathematical or routing complexity, but rather by standard high-level framework abstractions, rendering it highly viable for microsecond-level execution in optimized Edge AI systems.
